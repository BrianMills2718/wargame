# **Product Requirements Document: GenAI Geopolitical Wargame**

## **1. Product Vision & Executive Summary**
A hybrid natural-language wargaming platform that bridges the creativity of Large Language Models (LLMs) with the rigid consistency of a programmatic state engine. Unlike traditional wargames bound by hardcoded action menus, players issue high-level directives in natural language, delegate tasks to AI sub-agents, and manage scarce resources (attention, political capital, budget). 

The system relies on a **"Thin Canonical Spine"**—a fixed causal grammar and programmatic database that tracks concrete world states, hidden information, and dice rolls—while relying on a **Game Master LLM** to translate creative player actions into structured probability distributions for adjudication. The game concludes with an asymmetric scoring system evaluating how well players maximized their terminal values under their specific constraints.

---

## **2. Target Personas & Roles**
* **The Principal (Human Player):** Acts as the head of state (e.g., US President, Supreme Leader of Iran). Issues broad directives, allocates resources, and manages subordinates.
* **The Bureaucracy (LLM Sub-Agents):** Prompt-driven NPCs (e.g., Intel Chief, Defense Minister) that execute the Principal's directives, manage specific domains, and filter noisy world data into intelligence reports.
* **The Game Master (GM LLM):** The semantic translation engine. Converts natural language actions into JSON adjudication packets (probability distributions, affected variables) without having access to the "God Mode" truth unless necessary.
* **The Engine (Programmatic Core):** A Python/Database backend. The absolute source of truth. Handles RNG, enforces turn sequence, tracks hidden vs. public state, and determines observability.
* **The Strategic Expert (LLM Scorer):** End-of-game evaluator. Scores players based on value realization, strategic coherence, and constraint management, rather than simple win/loss conditions.

---

## **3. The Ontology (The "Thin Spine")**
The game engine strictly tracks concrete state, while the LLMs handle psychology and strategy. The programmatic schema is limited to:

1.  **Values:** Terminal intrinsic goals (e.g., Regime Survival, Dignity, System Stability).
2.  **State Variables (Concrete):** Objective measurements (e.g., `sanctions_intensity`, `proxy_network_presence`, `carrier_location`).
3.  **Instruments (MIDFIELD):** Means of influence (Military, Intelligence, Diplomacy, Finance, Information, Economic, Legal, Development).
4.  **Actions:** Specific operational moves taken by players or sub-agents.
5.  **State Transitions:** Deltas applied to State Variables based on adjudicated Actions.
6.  **Observations:** Filtered, noisy data packets generated by the Engine and delivered to specific actors based on visibility rules.

---

## **4. Core Gameplay Loop**
1.  **Context Delivery:** The Engine generates an `ObservationPacket` based on the player's visibility rights and passes it to the Sub-Agents.
2.  **Intelligence Briefing:** Sub-Agents interpret the observations and brief the Principal (Player).
3.  **Command Phase:** The Principal issues natural-language directives and allocates resources/attention budgets to Sub-Agents or specific initiatives.
4.  **Action Parsing:** A parser LLM converts the directives into structured `ActionIntent` JSON objects.
5.  **Adjudication Prep:** The Engine passes the `ActionIntent`, relevant state variables, and interaction models to the GM LLM.
6.  **Probability Generation:** The GM LLM returns an `AdjudicationPacket` JSON detailing possible outcomes (e.g., Success: 40%, Partial: 40%, Failure: 20%), affected variables, and observability rules.
7.  **Resolution:** The Engine rolls RNG against the GM's probabilities, applies the resulting `StateTransition` to the canonical database, and prepares the next turn's observations.

---

## **5. Feature Requirements**

### **P0 (Minimum Viable Product)**
* **Database / State Manager:** Relational or Document DB to store canonical hidden state, public state, and side-specific observable state.
* **Structured Output Enforcer:** Middleware (e.g., Pydantic + OpenAI Structured Outputs) to strictly enforce the GM LLM's JSON schemas and prevent pipeline crashes.
* **Basic Adjudication Engine:** Python backend to parse GM probabilities, execute RNG, and update state variables.
* **Two-Player Chat Interface:** UI for human-to-human play with LLM mediation.

### **P1 (Core Enhancements)**
* **Sub-Agent Delegation System:** Prompts and memory scoping for dedicated NPC ministers (Intel, Military, Economy) who consume player resources to act.
* **Dynamic Variable Creation:** Protocol allowing the GM LLM to propose new `StateVariables` mid-game if a player attempts a highly creative, out-of-schema action.
* **Asymmetric Scoring Engine:** End-of-game LLM pipeline to evaluate performance against initial Value parameters.

### **P2 (Advanced Mechanics)**
* **Espionage & Hidden Traits Module:** Sub-agents and NPCs have hidden profiles (Loyalty, Competence, Susceptibility). Players can spend resources to uncover, flip, or deceive these agents.
* **Memory Condensation Pipeline:** Vector DB or automated summarization scripts to compress turn histories and prevent context window collapse.

---

## **6. Non-Functional Requirements**
* **JSON Reliability:** 99.9% parse success rate from the GM LLM to the Programmatic Core.
* **Latency:** Turn resolution (from Player Command to Sub-Agent Briefing) must resolve in under 15 seconds to maintain player engagement.
* **Context Management:** Total prompt payload to any LLM must not exceed 80% of its maximum context window, leaving buffer for generation and dynamic variables.

---

## **7. Risk & Mitigation Matrix**
| Risk | Impact | Mitigation Strategy |
| :--- | :--- | :--- |
| **LLM Output Hallucination (Type Errors)** | High (Engine Crash) | Use strict API-enforced JSON schemas. Implement automated retry logic if the engine fails to parse the `AdjudicationPacket`. |
| **"God-Moding" by Game Master** | High (Breaks Game Balance) | Hardcode probability ceilings in the GM prompt (e.g., "Max success for covert action is 65% unless overwhelming capability advantage exists"). |
| **Context Window Collapse** | Medium (Amnesia) | Implement a cascading summary system. Only pass the current canonical state and the summarized history of the last 3 turns to the GM LLM. |
| **Sub-Agent Sycophancy** | Medium (Poor Gameplay) | System prompts for Sub-Agents must explicitly instruct them to act as "Red Teams," highlighting risks, failures, and dissenting opinions. |
